# リマインダー自然言語操作 設計書

## 概要
リマインダーの操作をID番号ベースから自然言語ベースに移行し、
通知後のスヌーズ（エスカレーション間隔）と会話文脈による操作を実現する。

## 現状の課題

1. **通知後の放置**: 通知が1回きりで、完了報告がなくても追跡しない
2. **ID番号での操作**: 「リマインダー5番を完了にして」はプログラム的で不自然
3. **文脈の欠如**: 「それ明日やる」の「それ」が解決できない

## 設計判断（確定）

### 1. スヌーズ方式
**採用: エスカレーション間隔**
- 通知後、完了報告がなければ間隔を徐々に広げて再通知
- ユーザーが鬱陶しくならない程度にリマインド

| 回数 | 間隔 | 累計経過 |
|------|------|----------|
| 初回通知 | — | 0 |
| 1回目スヌーズ | 30分後 | 30分 |
| 2回目 | 1時間後 | 1.5時間 |
| 3回目 | 3時間後 | 4.5時間 |
| 4回目以降 | 6時間ごと | — |

- スヌーズ通知は初回より簡潔にする（「まだ未完了: 〇〇」程度）
- ユーザーが「1週間後に言って」等と明示した場合はその時刻に再設定

### 2. 自然言語操作
**採用: LLMによる文脈解決 + 聞き返しフォールバック**

#### 解決フロー
```
[ユーザー発言: 「終わったよ」「それ明日やる」等]
  → SkillRouter → reminder unit
  → LLM に以下を注入:
      - ユーザーの発言
      - conversation_context（直近の会話履歴）
      - 通知済み未完了リマインダー一覧
      - 全アクティブリマインダー一覧
  → LLM が対象リマインダーと操作を推定

[推定結果]
  → 確信度高（1件に特定）→ 即実行
  → 曖昧（複数候補）→ 候補を提示して聞き返し
  → 不明 → 「どのリマインダーのことですか？」と聞き返し
```

#### 対応する自然言語パターン
| 発話例 | 解釈 |
|--------|------|
| 「終わったよ」「できた」 | 直近通知のリマインダーを完了 |
| 「買い物終わった」 | 「買い物」に該当するリマインダーを完了 |
| 「明日やる」「明日にして」 | 対象リマインダーを明日にリスケ |
| 「1週間後に言って」 | 対象リマインダーを1週間後にスヌーズ |
| 「もういい」「いらない」 | 対象リマインダーを削除 |
| 「時間変えて、15時にして」 | 対象リマインダーの時刻を変更 |

### 3. ID操作の扱い
**IDによる操作も内部的には維持する（後方互換）。**
ただし、通知メッセージからは番号による指示の案内を削除する。

現在の通知:
> リマインド: 買い物に行く
> 完了したら「リマインダー5番を完了にして」と教えてください。

変更後の通知:
> リマインド: 買い物に行く
> 終わったら教えてね！

## DBスキーマ変更

```sql
ALTER TABLE reminders ADD COLUMN snooze_count INTEGER NOT NULL DEFAULT 0;
ALTER TABLE reminders ADD COLUMN last_snoozed_at DATETIME;
ALTER TABLE reminders ADD COLUMN snoozed_until DATETIME;  -- 明示的スヌーズ指定時
```

| カラム | 用途 |
|--------|------|
| `snooze_count` | スヌーズ回数（エスカレーション間隔の算出に使用） |
| `last_snoozed_at` | 最後にスヌーズ通知した時刻 |
| `snoozed_until` | 「1週間後に言って」等で明示指定された次回通知時刻 |

## LLMプロンプト変更

### 現在の _EXTRACT_PROMPT への追加

```
## アクション一覧（追加）
- snooze: リマインダーを指定時刻まで延期（time が必要）
- contextual_done: 会話文脈から対象を推定して完了
- contextual_snooze: 会話文脈から対象を推定して延期（time が必要）
- ask_clarify: 対象リマインダーが特定できない場合

## 追加コンテキスト
### 通知済み未完了リマインダー（直近通知順）
{notified_reminders}

### 全アクティブリマインダー
{active_reminders}

## 推定ルール
- ユーザーの発言と会話履歴から、どのリマインダーについて話しているか推定してください
- 直近に通知されたリマインダーを優先的に候補とする
- 確信が持てない場合は action: "ask_clarify" を返し、candidates に候補IDリストを含めてください
- 「終わったよ」「できた」等の完了表現があれば action: "contextual_done"
- 「明日やる」「〇日後に言って」等の延期表現があれば action: "contextual_snooze"

## 出力形式（追加フィールド）
{"action": "...", "target_id": 数値, "candidates": [数値], "time": "YYYY-MM-DD HH:MM", "reason": "推定理由"}
```

## 処理フロー

### スヌーズ通知（ハートビート）

```
[on_heartbeat]
  → 通知済み（notified=1）かつ未完了（active=1）のリマインダーを取得
  → 各リマインダーについて:
      → snoozed_until が設定されている場合:
          → 現在時刻 < snoozed_until → スキップ
          → 現在時刻 >= snoozed_until → 再通知、snoozed_until をクリア
      → snoozed_until が未設定の場合:
          → snooze_count からエスカレーション間隔を算出
          → 現在時刻 - last_snoozed_at < 間隔 → スキップ
          → 間隔経過 → スヌーズ通知を送信
              → snooze_count += 1
              → last_snoozed_at = now
```

### 自然言語完了

```
[ユーザー: 「買い物終わった」]
  → SkillRouter → reminder unit
  → _extract_params に通知済みリマインダー一覧を注入
  → LLM: {"action": "contextual_done", "target_id": 5, "reason": "「買い物」が #5 の内容と一致"}
  → リマインダー #5 を完了
  → 「買い物のリマインダー、完了にしたよ！」
```

### 聞き返し

```
[ユーザー: 「終わったよ」（通知済みリマインダーが複数）]
  → LLM: {"action": "ask_clarify", "candidates": [5, 8]}
  → 「どれのこと？」
      #5 買い物に行く（14:00通知済み）
      #8 レポート提出（15:30通知済み）
  → ユーザー: 「買い物」
  → LLM: {"action": "contextual_done", "target_id": 5}
  → 完了処理
```

### 明示的スヌーズ

```
[ユーザー: 「1週間後に言って」]
  → LLM: {"action": "contextual_snooze", "target_id": 5, "time": "2026-04-15 14:00"}
  → snoozed_until = "2026-04-15T14:00:00"
  → snooze_count リセット
  → 「買い物のリマインダー、4/15 14:00 にまた通知するね」
```
